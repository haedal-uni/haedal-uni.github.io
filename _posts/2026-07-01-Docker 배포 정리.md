# Django 프로젝트 Docker 배포 정리

<br><br>

## 1. 서버 접속하기

로컬 PC에서 서버에 SSH로 접속한다.

```bash
ssh 사용자id@접속IP
```

<br>

ex)

```bash
ssh ubuntu@123.123.123.123
```

접속하면 서버 안에서 프로젝트 폴더를 만들거나, 기존 프로젝트 폴더로 이동해서 작업하면 된다.

<br><br><br><br>

## 2. 현재 서버에서 실행 중인 포트 확인하기

서버에서 이미 사용 중인 포트를 확인한다.

```bash
netstat -ntlp
```

<br>

또는 아래 명령어를 사용할 수도 있다.

```bash
sudo lsof -i -P -n | grep LISTEN
```

<br>

이 명령어를 실행하면 현재 실행 중인 포트번호를 확인할 수 있다.

예를 들어 `12345`번 포트를 사용할 예정인데 이미 다른 프로그램이 사용 중이라면 Docker 실행 시 다른 포트를 사용해야 한다.

```bash
docker run -d -p 12346:8000 --env-file .env --name 컨테이너명 이미지명
```

<br>

ex)

```bash
docker run -d -p 12345:8000 --env-file .env --name chat-container my-django-app
```

<br>

포트 의미는 다음과 같다.

```text
-p 12345:8000
```

```text
12345 → 서버 외부에서 접속할 포트
8000  → 컨테이너 내부 Django 서버 포트
```

즉, 위처럼 실행했다면 브라우저에서는 아래 주소로 접속한다.

```text
http://서버IP:12345
```

<br>

브라우저에서 접속하는 포트는 `-p` 옵션의 왼쪽 포트다.

```text
-p 12345:8000
   ↑     ↑
   │     컨테이너 내부 포트
   브라우저에서 접속할 포트
```

<br><br><br><br>

## 3. 프로젝트 폴더 구조 확인하기

Django 프로젝트는 보통 아래와 같은 구조가 되어야 한다.

```text
project/
├── chat/
├── config/
├── source/
├── .env
├── Dockerfile
├── manage.py
└── requirements.txt
```

`Dockerfile`, `manage.py`, `requirements.txt`, `.env` 파일이 같은 프로젝트 폴더 안에 있어야 한다.

<br>

예를 들어 `project` 폴더 안에 `manage.py`가 있다면 반드시 `project` 폴더 안에서 Docker 이미지를 빌드해야 한다.

현재 위치는 아래 명령어로 확인한다.

```bash
pwd
```

<br>

현재 폴더의 파일은 아래 명령어로 확인한다.

```bash
ls
```

<br>

아래 파일들이 보여야 한다.

```text
Dockerfile
manage.py
requirements.txt
.env
```

<br><br><br><br>

## 4. requirements.txt 작성하기

Dockerfile에서 아래 명령어로 패키지를 설치하기 때문에 `requirements.txt` 파일이 필요하다.

```dockerfile
RUN pip install --no-cache-dir -r requirements.txt
```

<br>

`pyproject.toml`을 사용하는 프로젝트라면 `pyproject.toml`의 `dependencies` 항목을 보고 `requirements.txt`에 작성하면 된다.

<br>

만약 로그에 아래와 같은 오류가 나온다면:

```text
ModuleNotFoundError: No module named 'pymysql'
```

`requirements.txt`에 `pymysql`이 없거나, `requirements.txt` 수정 후 Docker 이미지를 다시 빌드하지 않은 것이다.

이 경우에는 이미지를 다시 빌드해야 한다.

<br>

```bash
docker rm -f chat-container
docker build -t my-django-app .
docker run -d --name chat-container --add-host=host.docker.internal:host-gateway -p 12345:8000 --env-file .env -v "$(pwd):/app" my-django-app
```

<br><br><br><br>

## 5. Dockerfile 만들기

프로젝트 폴더 안에 `Dockerfile` 파일을 만든다.

```text
Dockerfile
```

<br>

```dockerfile
FROM python:3.11-slim

# 컨테이너 내 작업 디렉토리 설정
WORKDIR /app

# requirements.txt 복사 후 패키지 설치
COPY requirements.txt /app/
RUN pip install --no-cache-dir -r requirements.txt

# 프로젝트 전체 파일 복사
COPY . /app/

# Django 개발 서버 실행
# 컨테이너 내부에서는 8000번 포트로 실행
CMD ["python", "manage.py", "runserver", "0.0.0.0:8000"]
```

<br>

이 Dockerfile의 의미는 다음과 같다.

<br>

```dockerfile
FROM python:3.11-slim
```

Python 3.11이 설치된 가벼운 Linux 이미지를 기반으로 사용한다.

<br>

```dockerfile
WORKDIR /app
```

컨테이너 내부의 작업 위치를 `/app`으로 설정한다.

<br>

```dockerfile
COPY requirements.txt /app/
RUN pip install --no-cache-dir -r requirements.txt
```

`requirements.txt`를 컨테이너 안으로 복사한 뒤 필요한 Python 패키지를 설치한다.

<br>

```dockerfile
COPY . /app/
```

현재 프로젝트 폴더의 모든 파일을 컨테이너 내부 `/app`으로 복사한다.

<br>

```dockerfile
CMD ["python", "manage.py", "runserver", "0.0.0.0:8000"]
```

컨테이너가 실행되면 Django 서버를 실행한다.

<br><br><br><br>

## 6. settings.py 수정하기

Django는 기본적으로 허용된 호스트에서만 접속할 수 있다.

따라서 서버 IP를 `ALLOWED_HOSTS`에 추가해야 한다.

```python
# config/settings.py

ALLOWED_HOSTS = ['서버접속IP', 'localhost', '127.0.0.1']
```

<br>

*운영 환경에서는 필요한 도메인 또는 서버 IP만 명시하는 것이 좋다.

<br><br><br><br>

## 7. MariaDB 연결용 .env 작성하기

Django 컨테이너에서 서버에 설치된 DB에 연결하려면 `.env`를 아래처럼 작성한다.

```env
DB_NAME=user_db
DB_USER=user
DB_PASSWORD=비밀번호입력
DB_HOST=host.docker.internal
DB_PORT=3306
```

<br>

Docker 컨테이너에서 서버 DB에 연결할 것이기 때문에 `DB_HOST=localhost`가 아니라 `DB_HOST=host.docker.internal`로 작성했다.

내부에서 `localhost`는 MariaDB가 설치된 서버가 아니라 컨테이너 자기 자신을 의미한다.

<br>

Docker 실행 시 아래 옵션도 같이 넣어야 한다.

```bash
--add-host=host.docker.internal:host-gateway
```

<br>

전체 Docker 실행 예시는 다음과 같다.

```bash
docker run -d --name chat-container --add-host=host.docker.internal:host-gateway -p 12345:8000 --env-file .env -v "$(pwd):/app" my-django-app
```

<br>

비밀번호에 `#`이 들어가면 `.env`에서 주석처럼 처리될 수 있으므로 가능하면 피하는 것이 좋다. ex) `#1234`

<br><br><br><br>

## 8. MariaDB 데이터베이스와 사용자 생성하기

MariaDB에 root 계정으로 접속한다.

```bash
mysql -u root -p mysql
```

<br>

또는 아래처럼 접속해도 된다.

```bash
mysql -u root -p
```

<br>

MariaDB 접속 후 아래 SQL을 실행한다.

```sql
CREATE DATABASE IF NOT EXISTS user_db CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;

CREATE USER IF NOT EXISTS 'user'@'localhost' IDENTIFIED BY '비밀번호입력';
CREATE USER IF NOT EXISTS 'user'@'172.17.%' IDENTIFIED BY '비밀번호입력';

GRANT ALL PRIVILEGES ON user_db.* TO 'user'@'localhost';
GRANT ALL PRIVILEGES ON user_db.* TO 'user'@'172.17.%';

FLUSH PRIVILEGES;
```

<br>

각 SQL의 의미는 다음과 같다.

<br>

```sql
CREATE DATABASE IF NOT EXISTS user_db CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;
```

`user_db`라는 데이터베이스를 만든다.

`utf8mb4`는 한글과 이모지 등을 안정적으로 저장하기 위한 문자셋이다.

<br>

```sql
CREATE USER IF NOT EXISTS 'user'@'localhost' IDENTIFIED BY '비밀번호입력';
```

MariaDB 서버 내부에서 직접 접속하는 `user` 계정을 만든다.

<br>

```sql
CREATE USER IF NOT EXISTS 'user'@'172.17.%' IDENTIFIED BY '비밀번호입력';
```

Docker 컨테이너에서 접속하는 `user` 계정을 만든다.

Docker 기본 bridge 네트워크에서는 컨테이너 IP가 보통 `172.17.0.2` 같은 형태로 잡히기 때문에 `172.17.%`로 허용한다.

<br>

```sql
GRANT ALL PRIVILEGES ON user_db.* TO 'user'@'172.17.%';
```

`172.17.x.x` 대역에서 접속하는 `user` 계정에게 `user_db` 데이터베이스의 모든 테이블에 대한 권한을 준다.

<br>

아래처럼 정확한 컨테이너 IP만 허용하는 방식은 주의해야 한다.

```sql
CREATE USER IF NOT EXISTS 'user'@'172.17.0.2' IDENTIFIED BY '비밀번호입력';
GRANT ALL PRIVILEGES ON user_db.* TO 'user'@'172.17.0.2';
```

컨테이너를 삭제하고 다시 만들면 IP가 바뀔 수 있기 때문이다.

따라서 일반적으로는 아래처럼 Docker 대역만 허용하는 방식이 더 낫다.

```sql
'user'@'172.17.%'
```

<br>

전체 허용인 `%`는 개발 테스트에서는 편하지만, 배포 환경에서는 권장하지 않는다.

```sql
'user'@'%'
```

`%`는 어떤 IP에서든 접속을 허용한다는 뜻이므로, 배포 환경에서는 필요한 범위만 허용하는 것이 좋다.

<br><br><br><br>

## 9. MariaDB 사용자와 권한 확인하기

MariaDB에서 `user` 계정이 잘 만들어졌는지 확인한다.

```sql
SELECT user, host FROM mysql.user WHERE user = 'user';
```

정상이라면 아래처럼 나온다.

```text
+------+-----------+
| User | Host      |
+------+-----------+
| user | 172.17.%  |
| user | localhost |
+------+-----------+
```

<br><br> 

Docker 컨테이너 접속용 계정의 권한을 확인한다.

```sql
SHOW GRANTS FOR 'user'@'172.17.%';
```

정상이라면 아래처럼 나온다.

```text
GRANT ALL PRIVILEGES ON `user_db`.* TO `user`@`172.17.%`
```

<br>

비밀번호를 변경해야 할 경우에는 아래처럼 실행한다.

```sql
ALTER USER 'user'@'localhost' IDENTIFIED BY '새비밀번호';
ALTER USER 'user'@'172.17.%' IDENTIFIED BY '새비밀번호';

FLUSH PRIVILEGES;
```

<br>

이 경우 `.env`의 `DB_PASSWORD`도 같은 값으로 수정해야 한다.

```env
DB_PASSWORD=새비밀번호
```

## 10. Docker 이미지 빌드하기

Docker 이미지는 프로젝트 폴더 안에서 빌드해야 한다.

```bash
docker build -t 이미지명 .
```

<br>

ex)

```bash
docker build -t my-django-app .
```

여기서 마지막의 `.`은 현재 폴더를 기준으로 Docker 이미지를 만들겠다는 의미이다.

따라서 반드시 `Dockerfile`, `manage.py`, `requirements.txt`가 있는 폴더에서 실행해야 한다.

<br>

`requirements.txt`나 `Dockerfile`을 수정했다면 이미지를 다시 빌드해야 한다.

```bash
docker rm -f chat-container
docker build -t my-django-app .
docker run -d --name chat-container --add-host=host.docker.internal:host-gateway -p 12345:8000 --env-file .env -v "$(pwd):/app" my-django-app
```

<br><br><br><br>

## 11. Docker 컨테이너 실행하기

이미지를 빌드했다면 컨테이너를 실행한다.

일반 실행:

```bash
docker run -d --name chat-container --add-host=host.docker.internal:host-gateway -p 12345:8000 --env-file .env my-django-app
```

<br>

개발 중 코드 수정 반영을 편하게 하려면 볼륨 마운트 방식으로 실행한다.

```bash
docker run -d --name chat-container --add-host=host.docker.internal:host-gateway -p 12345:8000 --env-file .env -v "$(pwd):/app" my-django-app
```

<br>

이미 같은 이름의 컨테이너가 있다면 먼저 삭제해야 한다.

```bash
docker rm -f chat-container
```

그 후 다시 실행한다.

```bash
docker run -d --name chat-container --add-host=host.docker.internal:host-gateway -p 12345:8000 --env-file .env -v "$(pwd):/app" my-django-app
```

<br>

한 번에 실행하려면 아래처럼 작성한다.

```bash
docker rm -f chat-container
docker run -d --name chat-container --add-host=host.docker.internal:host-gateway -p 12345:8000 --env-file .env -v "$(pwd):/app" my-django-app
```

<br>

브라우저에서는 아래처럼 접속한다.

```text
http://서버IP:12345
```

<br><br><br><br>

## 12. 볼륨 마운트로 실행하기

Docker 이미지는 기본적으로 빌드 시점의 파일을 컨테이너 안에 복사한다.

따라서 코드를 수정하면 다시 빌드해야 반영된다.

하지만 개발 중에는 아래처럼 `-v` 옵션을 사용하면 현재 프로젝트 폴더를 컨테이너 내부 `/app`과 연결할 수 있다.

```bash
-v "$(pwd):/app"
```

이렇게 하면 파일 수정, 파일 추가, 파일 삭제가 컨테이너에 바로 반영된다.

<br>

볼륨 마운트 실행 명령어:

```bash
docker rm -f chat-container
docker run -d --name chat-container --add-host=host.docker.internal:host-gateway -p 12345:8000 --env-file .env -v "$(pwd):/app" my-django-app
```

<br>

단, `requirements.txt`에 패키지를 추가했을 때는 이미지를 다시 빌드해야 한다.

```bash
docker rm -f chat-container
docker build -t my-django-app .
docker run -d --name chat-container --add-host=host.docker.internal:host-gateway -p 12345:8000 --env-file .env -v "$(pwd):/app" my-django-app
```

<br>

주의할 점은 `.env` 파일이다.

`.env` 파일은 컨테이너를 만들 때 `--env-file .env` 옵션으로 한 번 읽힌다.

따라서 `.env`를 수정했다면 단순히 아래 명령어만 실행하는 것으로는 부족할 수 있다.

```bash
docker restart chat-container
```

`.env`를 수정한 경우에는 컨테이너를 삭제하고 다시 실행하는 것이 확실하다.

```bash
docker rm -f chat-container
docker run -d --name chat-container --add-host=host.docker.internal:host-gateway -p 12345:8000 --env-file .env -v "$(pwd):/app" my-django-app
```

<br><br><br><br>

## 13. 실행 중인 Docker 컨테이너 확인하기

컨테이너가 정상적으로 실행 중인지 확인한다.

```bash
docker ps
```

실행 중이라면 컨테이너 목록에 `chat-container`가 보여야 한다.

<br>

`PORTS` 부분에 아래처럼 보여야 한다.

```text
0.0.0.0:12345->8000/tcp
```

<br>

만약 아무것도 나오지 않는다면 컨테이너가 실행되었다가 바로 꺼졌을 가능성이 있다.

이 경우 꺼진 컨테이너까지 전부 확인한다.

```bash
docker ps -a
```

<br>

컨테이너가 꺼져 있다면 로그를 확인한다.

```bash
docker logs chat-container
```

최근 로그만 보고 싶으면 아래처럼 실행한다.

```bash
docker logs --tail 100 chat-container
```

<br><br><br><br>

## 14. Docker 로그 확인하기

컨테이너가 왜 꺼졌는지, Django 서버가 정상 실행되었는지 확인하려면 로그를 확인한다.

```bash
docker logs chat-container
```

<br>

실시간 로그를 보고 싶다면 아래처럼 실행한다.

```bash
docker logs -f chat-container
```

<br>

최근 100줄만 확인하고 싶다면 아래처럼 실행한다.

```bash
docker logs --tail 100 chat-container
```

## 15. Django 마이그레이션 실행하기

컨테이너가 실행 중이라면 Django migration을 실행한다.

```bash
docker exec -it chat-container python manage.py makemigrations
docker exec -it chat-container python manage.py migrate
```

<br>

`makemigrations`는 `models.py` 변경사항을 migration 파일로 만드는 단계이다.

```bash
docker exec -it chat-container python manage.py makemigrations
```

<br>

아래처럼 나오면 모델 변경사항이 없다는 뜻이다.

```text
No changes detected
```

<br><br> 

`migrate`는 migration 파일을 실제 DB에 적용하는 단계이다.

```bash
docker exec -it chat-container python manage.py migrate
```

<br>

아래처럼 나오면 DB 연결과 migration 상태가 정상이라는 뜻이다.

```text
Operations to perform:
  Apply all migrations: admin, auth, chat, contenttypes, sessions
Running migrations:
  No migrations to apply.
```

<br><br>  

정리하면 다음과 같다.

```text
makemigrations
→ models.py 변경사항을 migration 파일로 만드는 단계

migrate
→ migration 파일을 실제 DB에 적용하는 단계
```

## 16. Django 관리자 계정 만들기

migration이 정상적으로 끝난 뒤 관리자 계정을 만든다.

```bash
docker exec -it chat-container python manage.py createsuperuser
```

<br>

관리자 계정은 Docker 컨테이너가 아니라 DB에 저장된다.

따라서 DB가 유지되면 컨테이너를 삭제하고 다시 만들어도 관리자 계정은 다시 만들 필요가 없다.

<br>

관리자 계정이 이미 있는지 확인하려면 Django shell에 들어간다.

```bash
docker exec -it chat-container python manage.py shell
```

<br>  

그 다음 아래 코드를 실행한다.

```python
from django.contrib.auth.models import User
User.objects.filter(is_superuser=True).exists()
```

<br>

결과가 아래처럼 나오면 관리자 계정이 이미 있는 것이다.

```python
True
```

<br>

`False`인 경우에만 아래 명령어로 관리자 계정을 만들면 된다.

```bash
docker exec -it chat-container python manage.py createsuperuser
```

## 18. 수정한 내용별 해야 할 작업

| 수정한 내용 | 해야 할 것 |
|---|---|
| `views.py`, `urls.py`, `chatbot_graph.py`, 일반 Python 코드 | 보통 자동 반영 또는 `docker restart chat-container` |
| `templates/*.html`, CSS, JS | 브라우저 새로고침, 안 되면 강력 새로고침 |
| `.env` | 컨테이너 삭제 후 다시 실행 권장 |
| `settings.py` | `docker restart chat-container` 권장 |
| `models.py` | `makemigrations` + `migrate` 필요 |
| `requirements.txt` | `docker build` 다시 필요 |
| `Dockerfile` | `docker build` 다시 필요 |

<br>

코드 수정 후 재시작만 하고 싶을 때:

```bash
docker restart chat-container
```

<br>

`models.py`를 수정했을 때:

```bash
docker exec -it chat-container python manage.py makemigrations
docker exec -it chat-container python manage.py migrate
```

<br>

`.env`를 수정했을 때:

```bash
docker rm -f chat-container
docker run -d --name chat-container --add-host=host.docker.internal:host-gateway -p 12345:8000 --env-file .env -v "$(pwd):/app" my-django-app
```

<br>

`requirements.txt`를 수정했을 때:

```bash
docker rm -f chat-container
docker build -t my-django-app .
docker run -d --name chat-container --add-host=host.docker.internal:host-gateway -p 12345:8000 --env-file .env -v "$(pwd):/app" my-django-app
```

<br>

`Dockerfile`을 수정했을 때:

```bash
docker rm -f chat-container
docker build -t my-django-app .
docker run -d --name chat-container --add-host=host.docker.internal:host-gateway -p 12345:8000 --env-file .env -v "$(pwd):/app" my-django-app
```

<br>

정리하면 다음과 같다.

```text
코드만 수정함
→ docker restart 정도로 충분하거나 자동 반영

requirements.txt 수정함
→ docker build 다시 해야 함

models.py 수정함
→ makemigrations + migrate 필요

DB가 유지됨
→ createsuperuser 다시 안 해도 됨

DB가 새로 만들어짐
→ createsuperuser 다시 필요
```

## 19. 오류 1: manage.py 파일을 찾을 수 없는 경우

```bash
pwd
```

<br>

ex)

```bash
/home/ubuntu
```

<br>

만약 현재 위치가 홈 디렉토리라면 프로젝트 폴더로 이동해야 한다.

```bash
cd project
```

그다음 다시 빌드한다.

```bash
docker build -t my-django-app .
```

정리하면 아래 파일들이 있는 위치에서 빌드해야 한다.

```text

manage.py
requirements.txt
.env
```

## 22. 오류 4: DB 접속 권한 오류

로그에 아래 오류가 나오면:

```text
django.db.utils.OperationalError: (1045, "Access denied for user 'user'@'172.17.0.2' (using password: YES)")
```

Docker 컨테이너에서 DB로 접속하는 계정 권한이 없거나, 비밀번호가 맞지 않는 것이다.

<br>

MariaDB에서 아래를 확인한다.

```sql
SELECT user, host FROM mysql.user WHERE user = 'user';
```

<br>

`user | 172.17.%`가 없다면 추가한다.

```sql
CREATE USER IF NOT EXISTS 'user'@'172.17.%' IDENTIFIED BY '비밀번호입력';
GRANT ALL PRIVILEGES ON user_db.* TO 'user'@'172.17.%';
FLUSH PRIVILEGES;
```

권한 확인:

```sql
SHOW GRANTS FOR 'user'@'172.17.%';
```

<br>

정상이라면 아래처럼 나온다.

```text
GRANT ALL PRIVILEGES ON `user_db`.* TO `user`@`172.17.%`
```
